# 1. Executive Direction

This course is not a collection of computer tutorials.

It is a complete beginner transformation designed to help a person go from:

> “I am afraid I will break something.”

to:

> “I understand how computers work, I can use my own computer confidently, I can protect myself online, organize my digital life, solve common problems, and continue learning independently.”

The course must feel like a real academy program with:

- a clear beginning, middle, and end;
- progressive skill development;
- guided practice;
- platform-aware instruction;
- visible progress;
- practical assessments;
- realistic projects;
- confidence milestones;
- a final capstone;
- a meaningful certificate.

Do not build it as a large encyclopedia.  
Do not build it as disconnected articles.  
Do not build it as “click here, then click there.”  
Do not pad it with unrelated software.

The student must feel that every lesson moves them toward digital independence.

# 2. Product Positioning

## 2.1 Course promise

> Learn how computers and the internet really work, choose either Windows or Mac, practice essential everyday skills, protect yourself online, organize your files, communicate confidently, and solve common problems—step by step in Haitian Creole.

## 2.2 Student transformation

By the end of the course, the learner should be able to:

1. Explain the basic parts of a computer and what each part does.
2. Understand the relationship between hardware, an operating system, applications, files, the internet, websites, and cloud services.
3. Use either a Windows computer or a Mac confidently.
4. Navigate the desktop, launch and switch applications, manage windows, and change important settings.
5. Type, select, copy, paste, undo, use shortcuts, and enter Haitian Creole and French accented characters.
6. Create, name, organize, find, move, copy, compress, back up, and delete files and folders.
7. Connect safely to Wi-Fi and understand routers, mobile data, hotspots, browsers, search engines, websites, downloads, and uploads.
8. Search the web intelligently and evaluate whether information is trustworthy.
9. Create and manage email, attachments, contacts, and professional messages.
10. Use a browser and common web applications without confusing the browser, website, account, and internet.
11. Install, update, and remove legitimate applications safely.
12. Connect common devices such as printers, headphones, USB drives, external storage, phones, and webcams.
13. Protect accounts, devices, personal information, and money from common scams.
14. Back up important files and photos.
15. Perform basic maintenance and solve common computer and internet problems.
16. Evaluate a new or used computer before buying it.
17. Use AI as a learning assistant without depending on it blindly.
18. Complete realistic tasks for school, work, family, and daily life.
19. Know when a problem is safe to solve personally and when professional help is needed.
20. Be ready for a specialized next course such as Microsoft Office for Work or Computer Technician & Repair.

## 2.3 What makes this course different

Most beginner courses teach software menus or isolated tasks. This course teaches a reusable mental model first.

The recurring model is:

**What is it? → Why does it exist? → How does it work? → When is it useful? → How do I do it on my computer? → Can I complete a real task with it?**

The product is not “information.”

The product is **digital confidence demonstrated through action**.

# 5. Platform Strategy: Windows or Mac

## 5.1 Core decision

The learner selects a primary computer path during onboarding:

- **Windows Path**
- **Mac Path**

The student may switch paths at any time from course settings.

Do not create two separate courses.

Do not duplicate the entire lesson library.

The shared concept comes first, followed by the selected platform instructions.

## 5.2 Content model

Every lesson has one of four platform modes:

### Mode A — Universal

The concept and task are effectively the same on both platforms.

Examples:

- What is a computer?
- What is the internet?
- What is a file?
- How passwords work.
- How scams create urgency.
- What cloud storage means.

No unnecessary Windows/Mac tabs.

### Mode B — Shared concept with platform steps

The concept is shared, but the interface or keyboard command differs.

Examples:

- Copy and paste.
- Take a screenshot.
- Open settings.
- Search for a file.
- Install an application.
- Connect to Wi-Fi.
- Force-close a frozen application.

Show:

1. Shared explanation.
2. The learner’s selected platform.
3. A small “See the other platform” option.

### Mode C — Platform-specific lesson

The skill is unique enough to justify separate lesson variants.

Examples:

- File Explorer on Windows.
- Finder on Mac.
- Start menu and taskbar.
- Dock, menu bar, and Spotlight.
- Windows Update.
- macOS Software Update.

These variants share the same lesson ID family and learning outcome but contain distinct instructions.

### Mode D — Cross-device support

Android or iPhone appears only when the computer task requires it.

Examples:

- Transfer photos to a computer.
- Use a phone as a hotspot.
- Scan a QR code.
- Approve two-factor authentication.
- Send a file from a phone to a computer.
- Back up phone photos.

Do not give Android and iPhone equal space in every lesson.

## 5.3 Onboarding flow

Before Module 1:

1. Welcome screen.
2. “Do you have access to a computer?”
3. “Which computer will you use most?”
   - Windows
   - Mac
   - I am not sure
4. If unsure, show visual identification:
   - Apple logo / macOS desktop cues.
   - Windows logo / Start menu cues.
5. Save `preferredPlatform`.
6. Run a zero-stakes confidence check.
7. Explain that students can change paths later.
8. Download or open the practice-file pack.

## 5.4 Engineering requirement

Use one canonical lesson record with conditional content blocks.

Suggested shape:

```ts
type Platform = "universal" | "windows" | "mac" | "android" | "iphone";

type LessonBlock = {
  id: string;
  type:
    | "goal"
    | "explanation"
    | "diagram"
    | "vocabulary"
    | "platform_steps"
    | "practice"
    | "mission"
    | "mistake"
    | "safety"
    | "ai_help"
    | "knowledge_check"
    | "reflection";
  platforms?: Platform[];
  content: unknown;
};

type CourseLesson = {
  id: string;
  moduleId: string;
  order: number;
  title: string;
  estimatedMinutes: number;
  mode: "universal" | "shared_steps" | "platform_variant" | "cross_device";
  objectives: string[];
  blocks: LessonBlock[];
  requiredMission?: boolean;
  practiceFiles?: string[];
};
```

Never fork progress into two unrelated courses.

A student completing the Windows variant or Mac variant earns completion for the same learning outcome.

# 7. Learning Design System

## 7.1 Lesson rhythm

A standard lesson should follow this rhythm:

1. **Prepare** — connect the lesson to a real need.
2. **Understand** — explain the concept.
3. **See** — show a diagram, illustration, or interface.
4. **Do** — guide the selected platform steps.
5. **Practice** — let the learner try safely.
6. **Apply** — complete a real-life task.
7. **Check** — answer a short knowledge check.
8. **Reflect** — confirm what the learner can now do.

Do not force all eight into every tiny lesson, but every lesson must include understanding, action, and checking.

## 7.2 Required signature components

### Goal

A short outcome written as:

> After this lesson, you will be able to…

### Simple explanation

Use plain Haitian Creole. Explain new terms immediately.

### Digital Words

Introduce 3–5 essential terms where appropriate.

Each word requires:

- the term;
- a plain definition;
- a visual or example;
- pronunciation support only when useful;
- one sentence showing the term in real life.

Do not turn this into a dictionary dump.

### How It Works

A simple visual chain.

Example:

```text
Keyboard → Operating system → Application → Screen
```

or:

```text
Internet provider → Modem/router → Wi-Fi → Computer → Browser → Website
```

### Platform steps

Only after the learner understands the concept.

Steps must:

- use current interface names;
- avoid relying only on color or exact button position;
- tell the learner what result to expect;
- include a recovery note when the screen looks different;
- distinguish Windows and Mac accurately.

### Common mistake

Focus on beginner confusion, not trivia.

Examples:

- Browser vs. search engine.
- Save vs. Save As.
- Download vs. upload.
- Delete vs. uninstall.
- Wi-Fi vs. internet.
- Account password vs. device PIN.
- Closing a window vs. quitting an application.

### Safe practice

Provide an action the learner can perform without risking files, accounts, or money.

### Real-Life Mission

A practical task with:

- scenario;
- objective;
- required steps;
- success criteria;
- hint;
- platform notes;
- optional stretch challenge.

### Ask AI

Show one safe, specific way to use an AI assistant for the lesson.

Each AI box must also remind the learner:

- Do not share passwords, PINs, verification codes, bank details, private IDs, or sensitive files.
- Verify important instructions before changing settings or deleting data.
- AI may be wrong.

### Knowledge check

Use 2–5 questions.

Question types:

- multiple choice;
- select all that apply;
- order the steps;
- match term to meaning;
- identify the safer action;
- scenario decision.

Do not use trick questions.

### Confidence reflection

Examples:

- “I can do this alone.”
- “I can do it with a reminder.”
- “I need more practice.”

This is not graded but should inform review recommendations.

## 7.3 Tone

The course must be:

- respectful;
- calm;
- adult;
- warm;
- clear;
- encouraging;
- never childish;
- never condescending;
- never overly technical;
- never filled with generic motivational language.

Avoid:

- “Technology is magic!”
- “It’s super easy!”
- “Anyone can master everything instantly!”
- shame about not already knowing;
- jokes that make beginners feel behind.

Use:

> “This may feel unfamiliar at first. We will separate it into small actions.”

## 7.4 Language rules

Primary lesson language is Haitian Creole.

- Use natural Haitian Creole, not word-for-word English translation.
- Introduce the English interface term because the computer may display English.
- Show French interface terms only when they commonly help the audience.
- Keep button names exactly as they appear in the selected operating system.
- Explain abbreviations the first time.
- Avoid switching entire paragraphs into English.
- Do not translate brand names.

Recommended terminology pattern:

> **Folder (Dosye):** yon espas ou itilize pou òganize fichye.

Maintain a centralized terminology glossary so the same concept is translated consistently in every lesson.

# 8. Assessment System

## 8.1 Diagnostic confidence check

Before the course, ask the learner to perform or identify simple tasks.

This does not block access and does not produce a failing score.

Measure comfort with:

- mouse or trackpad;
- keyboard;
- opening an application;
- finding a file;
- connecting to Wi-Fi;
- using a browser;
- recognizing a suspicious message.

Use results to recommend optional review lessons.

## 8.2 Lesson checks

- 2–5 questions.
- Immediate feedback.
- Explain why each answer is right or wrong.
- Unlimited retries.
- No punitive score.

## 8.3 Module missions

Every module ends with a practical mission.

Module missions must test transfer, not memory.

Bad mission:

> List three types of files.

Good mission:

> Download the practice ZIP, extract it, rename the files, place them in the correct folders, and locate one file using search.

## 8.4 Confidence checkpoints

There are four larger checkpoints:

- Checkpoint 1 after Module 3.
- Checkpoint 2 after Module 6.
- Checkpoint 3 after Module 10.
- Checkpoint 4 after Module 14.

Each checkpoint combines multiple skills in one scenario.

Passing recommendation: 80%.  
Retry: unlimited, with targeted lesson recommendations.

## 8.5 Final exam

- 50 questions.
- Scenario-heavy.
- Covers all modules.
- Randomized from a larger bank.
- 80% required.
- Unlimited retries.
- Show category performance after each attempt.
- Do not reveal the entire answer bank.

## 8.6 Capstone

The capstone is more important than the final exam.

See Module 14 for the full capstone specification.

# 9A. Visual and Image Generation System

## 9A.1 Core principle

A lesson visual is an instructional object, not decoration.

Claude Code must not generate an image merely because a lesson has an image slot.

Before requesting an image, it must define what the visual teaches.

Every visual must answer at least one of these:

- What parts make up this system?
- What order does this process follow?
- What is the difference between two concepts?
- Where should the learner look?
- What mistake should the learner avoid?
- What does success look like?
- How does a real-life scenario connect to the lesson?

If the visual does not improve understanding, do not generate it.

## 9A.2 Use the correct visual type

### Use a real screenshot when:

- the learner must recognize an actual Windows or macOS interface;
- exact menu or setting names matter;
- the learner must follow platform steps.

Do not use AI-generated screenshots as factual interface instructions.

### Use a diagram when:

- teaching a process, hierarchy, relationship, flow, or comparison;
- showing internet, Wi-Fi, files, storage, email, cloud, backup, or troubleshooting concepts.

Prefer SVG, HTML/CSS, or another deterministic diagram format when accuracy matters.

### Use an annotated photo or realistic illustration when:

- identifying physical computer parts;
- showing ports, devices, workspaces, safe handling, or buying inspection;
- illustrating a real-life scenario.

### Use a simulation when:

- the learner needs to make a choice or practice an interface safely;
- feedback and interaction matter more than appearance.

### Use a decorative image only:

- at module openings;
- when it reinforces the emotional context without pretending to teach a factual interface;
- sparingly.

## 9A.3 OpenAI image API decision

Claude Code may use the OpenAI API for original illustrations when approved.

Important distinction:

- A ChatGPT subscription is not an API integration.
- Claude Code cannot “connect to ChatGPT” through the user’s ChatGPT account.
- The application must use an OpenAI API project and API key.
- The key must be stored server-side or in a secure build pipeline.
- Never expose the API key in client-side code, Git, generated lesson files, logs, screenshots, or browser requests.
- API usage has separate billing and rate limits.
- Use the currently supported GPT Image model documented by OpenAI rather than hard-coding an obsolete model name permanently.
- Keep the image provider behind an adapter so OpenAI, Gemini, or another approved provider can be changed later.

Suggested provider interface:

```ts
type CourseVisualRequest = {
  lessonId: string;
  visualId: string;
  visualType: "concept_diagram" | "realistic_illustration" | "scenario" | "object_identification";
  learningObjective: string;
  misconceptionToPrevent?: string;
  requiredElements: string[];
  forbiddenElements: string[];
  platform?: "universal" | "windows" | "mac";
  aspectRatio: string;
  brandStyleId: string;
  referenceAssetIds?: string[];
};

type CourseVisualResult = {
  provider: string;
  model: string;
  promptVersion: string;
  assetPath: string;
  generatedAt: string;
  reviewStatus: "pending" | "approved" | "rejected";
  reviewerNotes?: string;
};
```

## 9A.4 Provider strategy

Do not let one provider automatically generate every visual.

Recommended hierarchy:

1. Real screenshot for real UI.
2. Deterministic SVG/HTML diagram for factual systems.
3. Existing approved photograph or product image with rights verified.
4. OpenAI GPT Image for custom lesson illustrations and scene-based teaching visuals.
5. Gemini or another provider only when it passes the same review standard.

The failure in the current course is not only Gemini quality. It is the lack of a visual brief and approval process.

Before every generated image, Claude Code must create a sidecar visual brief:

```yaml
lesson_id:
visual_id:

student_should_notice:

```

Prompt generation must be based on this brief.

Do not use a vague prompt such as:

> Create an illustration about computer storage.

Use a precise instructional request such as:

> Create a clean, adult educational cutaway illustration comparing temporary working space and permanent storage. On the left, show a desk with several documents actively in use representing RAM. On the right, show a labeled filing cabinet containing saved folders representing SSD storage. A simple arrow shows that applications load from storage into working space. No robots, no glowing brain, no futuristic city, no random code, no decorative text, no inaccurate computer interface. Leave clear empty space below for a separately rendered caption.

## 9A.6 Text inside generated images

Default: no text inside AI-generated images.

Reasons:

- Haitian Creole spelling must remain exact;
- generated text may be incorrect;
- the same visual may be reused in another language;
- accessible HTML text is better.

Add labels through HTML/SVG overlays or post-production when possible.

If text must be generated:

- keep it minimal;
- supply exact text;
- review every character;
- reject the image for any spelling error.

Create one approved course visual style before mass generation.

## 9A.8 Visual QA rubric

Every generated visual must be reviewed on:

| Criterion | Requirement |
|---|---|
| Lesson relevance | Directly teaches the assigned objective |
| Concept accuracy | No factual or technical error |
| Required elements | All required objects/actions are visible |
| Distraction control | No irrelevant decorative objects |
| Device accuracy | Ports, hardware, and device behavior are plausible |
| Platform truth | Does not invent a Windows or Mac interface |
| Cultural fit | People and environment fit the audience respectfully |
| Brand consistency | Matches the approved visual system |
| Accessibility | Can be described clearly in alt text |
| Mobile readability | Main idea remains visible at small size |
| Text correctness | No misspelled or invented text |
| Originality | Does not imitate protected characters or another brand style |

Approval rule:

A visual must pass every critical accuracy category before it enters a lesson.

## 9A.9 Human review is mandatory

Claude Code may generate drafts automatically, but it must not auto-publish them.

Workflow:

1. Lesson objective approved.
2. Visual need identified.
3. Visual brief created.
4. Provider selected.
5. Draft generated.
6. Draft stored in a review folder.
7. Human reviewer approves, rejects, or requests a revision.
8. Approved asset is optimized and linked to the lesson.
9. Alt text is written from the final approved image.
10. Asset metadata records provider, prompt version, and review date.

## 9A.10 Cost and duplicate control

Before calling an image API:

- calculate a stable hash from the approved visual brief;
- check whether an approved matching asset already exists;
- do not regenerate unchanged visuals;
- set per-module generation limits;
- store failed/rejected generations separately;
- do not retry indefinitely;
- log estimated and actual provider cost where available.

# Module 12 — Buying and Caring for a Computer
**Purpose:** Help learners make financially sound decisions and extend device life.

## Learning outcomes

- evaluate new, used, and refurbished options;

## Lessons

### 104. Before You Buy Any Computer: Start With the Work
This is the required first lesson before brands or processors.

Teach the learner to define:

- the main jobs the computer must perform;
- school, office, browsing, video calls, business, coding, design, editing, or gaming needs;
- applications that must run;
- whether those applications require Windows or macOS;
- portability;
- battery expectations;
- screen-size preference;
- keyboard comfort;
- available repair and parts support;
- internet limitations;
- total budget including charger, bag, mouse, external storage, software, shipping, tax, and possible repair;
- how long the learner expects to keep the machine.

Required activity:

Create a personal **Computer Needs Profile** before seeing any recommendations.

Non-negotiable teaching rule:

> The “best computer” is not the most expensive machine. It is the machine that reliably performs the learner’s real work within the full budget.

### 105. Laptop, Desktop, All-in-One, Chromebook, or Mac?
Teach:

- laptop strengths and limitations;
- desktop value, upgradeability, and power requirements;
- all-in-one convenience and repair limitations;
- Chromebook strengths, internet dependence, storage limitations, and software compatibility;
- Mac as a computer family, not a device category separate from laptop/desktop;
- MacBook Air vs. MacBook Pro at a beginner level;
- Mac mini and iMac awareness;
- Windows laptop vs. Windows desktop;
- repairability and local parts availability;
- no blanket “Mac is better” or “Windows is better” conclusion.

Required comparison activity:

The learner chooses a device type for three fictional people and explains each choice.

### 106. Intel vs. AMD on Windows Computers
This lesson must be detailed enough to prevent bad purchases but remain beginner-friendly.

Teach:

- Intel and AMD are processor companies;
- processor brand alone does not determine quality;
- model family, generation, tier, cooling, power limits, RAM, SSD, and overall laptop design all affect performance;
- Intel Core families and modern naming awareness;
- AMD Ryzen families and modern naming awareness;
- entry, mainstream, and performance tiers;
- why “Core i7” or “Ryzen 7” without a generation/model number is incomplete;
- why an older high-tier processor may be slower or less suitable than a newer mid-tier processor;
- U/low-power vs. H/HS/HX performance-class awareness without requiring memorization;
- integrated graphics vs. dedicated graphics;
- battery and heat tradeoffs;
- application compatibility is usually broad on modern Windows computers, but specialized software should still be checked;
- benchmark numbers are evidence, not the only buying decision;
- avoid declaring one brand universally superior.

Required examples:

- Compare an old Intel Core i7 laptop with a newer Core i5 laptop.
- Compare an older Ryzen 7 with a newer Ryzen 5.
- Explain why the full processor name matters.
- Identify suspicious listings that show only “Intel,” “Core i7,” or “AMD” with no generation or exact model.

Durability rule:

Do not hard-code a permanent ranking of current chips. Teach the comparison method and place time-sensitive recommendations in a separately dated buying guide.

### 107. Understanding Processor Names and Generations
Teach the learner how to collect and interpret, at a practical level:

- exact processor name;
- manufacturer;
- product family;
- tier;
- generation or architecture indicator;
- suffix;
- number of cores only as one factor;
- device cooling and power behavior;
- supported operating system;
- release age;
- real workload suitability.

The learner does not need to decode every naming scheme from memory.

Required tool:

A processor-name worksheet that asks:

1. What is the complete name?
2. Approximately how old is it?
3. Is it intended for low power, mainstream use, or high performance?
4. Does it meet the application’s requirements?
5. Is the rest of the computer balanced?

### 108. Intel Mac vs. Apple Silicon Mac
Teach this before comparing M-series generations.

Explain:

- older Macs may use Intel processors;
- newer Macs use Apple silicon;
- in **About This Mac**, a Mac showing **Chip** followed by an M-series name uses Apple silicon;
- a Mac showing **Processor** followed by Intel is Intel-based;
- Apple silicon and Intel Macs may differ in performance, battery life, heat, software compatibility, startup/recovery behavior, Windows virtualization options, and long-term operating-system support;
- some older Intel applications may run on Apple silicon through translation technology, but compatibility must be checked;
- a cheap Intel Mac is not automatically a good value;
- an Intel Mac may still fit a narrow need, but the student must evaluate software support, battery condition, age, repair cost, and remaining updates;
- never recommend a used Mac only because it has an Apple logo.

### 109. Apple M1, M2, M3, M4, M5 and Future M-Series Chips
Teach the system, not a frozen shopping list.

Explain:

- M1, M2, M3, M4, M5, and later names represent Apple silicon generations;
- newer generation does not automatically mean the better purchase at every price;
- base, Pro, Max, and Ultra indicate different performance classes, not simple quality grades;
- a base newer chip may be better for an ordinary user than an older Pro/Max machine with poor battery, insufficient storage, damage, or excessive price;
- memory and storage are generally not user-upgradeable after purchase on Apple silicon Macs;
- unified memory is not marketed or evaluated exactly like a traditional Windows RAM module, but the learner still needs enough memory for the workload;
- check external-display support, ports, application support, and accessories for the exact model;
- distinguish chip generation from computer model and release year;
- compare total machine condition and configuration, not only the “M” number.

Required comparison cases:

- M1 MacBook Air vs. Intel MacBook Pro.
- Base M2/M3/M4/M5 machine vs. an older Pro-tier chip.
- 8 GB vs. 16 GB or more memory for different workloads.
- 256 GB vs. 512 GB or greater storage.
- Newer low-storage machine vs. older well-configured machine.
- A cheap Activation-Locked Mac that must be rejected.

Time-sensitive rule:

Maintain a separate `buying-guide-current.md` file with a visible **last verified date** for currently sold Apple models and market-specific recommendations. Do not rewrite the evergreen lesson every time Apple introduces a chip.

# 22. Final Product Test

Before approving the course, answer these questions honestly:

1. Can a complete beginner understand why a computer behaves the way it does?
2. Can a Windows student finish without being distracted by constant Mac instructions?
3. Can a Mac student complete the same learning outcomes without being treated as an afterthought?
4. Does every module end in a skill the learner can demonstrate?
5. Does the course remain useful after minor interface updates?
6. Does it work under slow or intermittent internet?
7. Does it respect adults?
8. Does it prepare students to protect their information and money?
9. Does it teach buying decisions that may save students from a bad purchase?
10. Can the learner organize and back up important files?
11. Can the learner communicate by email and handle attachments?
12. Can the learner use a repeatable troubleshooting process?
13. Does the capstone prove digital independence?
14. Does the certificate mean something?
15. Does the final student feel less dependent on other people?

If any answer is no, the course is not finished.

# 23. Final Instruction

Build **Computer & Internet Essentials** as a coherent educational product from onboarding to certificate.

The learner should not merely consume 132 lesson screens.

The learner should repeatedly understand, practice, succeed, reflect, and apply.

The final outcome is not:

> “I watched a computer course.”

The final outcome is:

> “I understand my computer. I can use it confidently. I can protect myself. I can solve common problems. I am ready for what comes next.”
